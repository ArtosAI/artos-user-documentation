This page walks through how a few essential actions are applied to real sections of a Clinical Study Report (CSR). These examples show what complete, production-ready configurations look like — the kind of specificity that produces consistent, high-quality output.

The examples below use a fictional compound (Compound X) and a standard Phase 3 CSR structure. The same configuration principles apply across document types.

---

## Copy from Source — Study Objectives

The Study Objectives section in a CSR should reflect the protocol language exactly. This is a verbatim lift, so Copy from Source is the right action.

| Component | Configuration |
| --- | --- |
| **Action** | Copy from Source |
| **Action Instructions** | "Copy the study objectives verbatim from the protocol. Include the primary objective and all secondary objectives as written. Do not paraphrase." |
| **Custom Instructions** | _(none — the section is entirely verbatim)_ |

**Why this works:** The protocol language is final and legally significant. There's no reason to rewrite it. Copy from Source preserves it exactly.

---

## Copy & Modify from Source — Study Design

The Study Design section comes from the protocol, but needs to be converted from future tense ("patients will be randomized") to past tense ("patients were randomized") since the study is complete.

| Component | Configuration |
| --- | --- |
| **Action** | Copy & Modify from Source |
| **Action Instructions** | "Copy the study design section from the protocol. Change all verbs to past tense to reflect that the study has been completed. Replace \[drug code\] with \[INN\] throughout." |
| **Custom Instructions** | _(none)_ |

**Why this works:** The structure and content of the protocol's study design section are correct — it just needs tense and terminology corrections. Copy & Modify makes those targeted changes without rewriting the section.

## Summarize from Source — Safety Results: Adverse Events

The Safety Results section requires synthesis, not verbatim copying. The source (the clinical database or listings) contains more detail than the CSR needs, so Summarize from Source is appropriate.

| Component | Configuration |
| --- | --- |
| **Action 1** | Summarize from Source |
| **Action 1 Instructions** | "Summarize adverse events from the clinical study report. Focus on events occurring in ≥5% of patients in any treatment group. Present serious adverse events before common adverse events. Include comparison to placebo where available." |
| **Action 2** | Use Template Table |
| **Action 2 Instructions** | "Populate the adverse events summary table. Include all events with ≥5% incidence in any treatment group. Sort by system organ class, then by descending frequency within each class." |
| **Custom Instructions** | "Present SAEs before common AEs. Limit the narrative to 2 pages. Do not repeat data from the table in the narrative." |

**Why this works:** The narrative and table serve different purposes. The Summarize action produces the interpretive text; the Template Table action populates the structured data. Custom Instructions ensure they work together without redundancy.

## Use Template Text — Regulatory Statements

Some sections contain standard regulatory language that doesn't change between studies — disclaimer language, reference to ICH guidelines, standard methodology statements.

| Component | Configuration |
| --- | --- |
| **Action** | Use Template Text |
| **Action Instructions** | "Use the standard regulatory statement as written in the template. Do not modify." |
| **Custom Instructions** | _(none)_ |

**Why this works:** When the content is identical across every document generated from this template, Use Template Text is the right action. There's nothing to look up, summarize, or extract.

## Use Template Table — Demographics

The Demographics section uses a table defined in the Word template. The structure (columns, headers) is fixed; only the data changes between studies.

| Component | Configuration |
| --- | --- |
| **Action** | Use Template Table |
| **Action Instructions** | "Populate the demographics and baseline characteristics table for the ITT population. Include: age (mean, SD, range), sex (n, %), race (n, %), weight (mean, SD), height (mean, SD), BMI (mean, SD). Present for each treatment group and overall." |
| **Custom Instructions** | "Present continuous variables as mean ± SD. Present categorical variables as n (%). Round percentages to one decimal place." |

**Why this works:** The table structure lives in the Word template and won't change. The action instruction specifies exactly which population, which variables, and how to format the values.

## Create Intra-Document Summary — Synopsis

The Synopsis summarizes the entire CSR into a standalone section. It draws from content already generated elsewhere in the document.

| Component | Configuration |
| --- | --- |
| **Action** | Create Intra-Document Summary |
| **Action Instructions** | "Create a comprehensive standalone synopsis covering: study design and objectives, patient population and disposition, primary and secondary efficacy results (with p-values and confidence intervals), safety overview (SAEs, deaths, discontinuations due to AEs), and conclusions. Limit to 1–1.5 pages." |
| **Custom Instructions** | "The synopsis should stand alone. A reader should be able to understand the study without reading the full report. Use quantitative data throughout. Present efficacy before safety." |

**Why this works:** The Synopsis needs to reflect the rest of the document, not the raw source files. Create Intra-Document Summary reads what Artos has already drafted and produces a coherent summary from it.

## Fill in the Blank — Cover Page

The cover page contains fixed text with specific placeholders: study title, compound name, protocol number, dates, sponsor name.

| Component | Configuration |
| --- | --- |
| **Action** | Fill in the Blank |
| **Action Instructions** | "Fill in all placeholders on the cover page. Use the protocol for \[study title\], \[compound name\] (use INN), \[protocol number\], \[sponsor name and address\], and \[phase\]. Use the CSR for \[data cut-off date\] and \[total number of patients enrolled\]." |
| **Custom Instructions** | _(none)_ |

**Why this works:** The cover page structure never changes. Only the study-specific values need to be populated. Fill in the Blank handles exactly this — filling fixed slots from source data without touching the surrounding text.

## Summarize TFLs — Primary Efficacy Results

When the primary efficacy analysis table comes from a formal TFL output (Tables, Figures, and Listings), use Summarize TFLs to insert the table and generate the accompanying narrative.

| Component | Configuration |
| --- | --- |
| **Action 1** | Summarize TFLs |
| **Action 1 Instructions** | "Insert the primary efficacy analysis table (Table 14.2.1). Provide a narrative describing the primary endpoint result: include the endpoint definition, analysis population, treatment effect with 95% CI, p-value, and clinical interpretation. State whether the primary endpoint was met." |
| **Action 2** | Summarize TFLs |
| **Action 2 Instructions** | "Insert the key secondary endpoint table (Table 14.2.2). Summarize each secondary endpoint result in order of the statistical testing hierarchy. Include point estimate, 95% CI, and p-value for each." |
| **Custom Instructions** | "Present primary endpoint results before secondary. Include confidence intervals in all results narratives. Note any multiplicity adjustments applied." |

**Why this works:** TFL documents are the authoritative statistical outputs. Summarize TFLs inserts the table as-is and generates interpretive text that explains what the numbers mean — combining data fidelity with narrative clarity.

## Generate from Listings — Patient Narratives

Patient Narratives require one writeup per subject who experienced a serious adverse event. Generate from Listings handles this automatically.

| Component | Configuration |
| --- | --- |
| **Action** | Generate from Listings |
| **Action Instructions** | "Generate a narrative for each subject who experienced a serious adverse event. For each subject include: demographics (age, sex, race), relevant medical history, study treatment (dose, duration, any modifications), SAE details (term, onset date/study day, severity, seriousness criteria, relationship to study drug, actions taken, outcome, resolution date). Present events in chronological order." |
| **Custom Instructions** | "Each narrative should be a standalone paragraph. Use past tense throughout. Refer to subjects by subject number only, not by name. Include all events for the subject even if multiple SAEs occurred." |

**Why this works:** The listings document contains one row per subject event. Generate from Listings reads those rows and produces a structured narrative per subject, applying the same instruction to each one. The Custom Instructions ensure consistent formatting across all narratives.
